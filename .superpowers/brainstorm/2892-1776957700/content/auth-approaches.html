How should login work?

Three approaches for Auth + Roles. Each trades security for simplicity differently.

A

Username + 4-digit PIN → JWT token Recommended

User picks a username, sets a PIN. App stores a JWT in localStorage. Backend validates the JWT on every request and knows who's calling. Token expires after 30 days (crew doesn't re-login constantly). Admin resets PINs from user management screen.

Pros

  • Backend knows who's calling
  • Real audit trail per user
  • PIN is crew-friendly
  • Token = stay logged in

Cons

  • Requires PIN hashing on backend
  • JWT middleware on all routes
B

Username + Password → JWT token

Same as A but full text password instead of PIN. More secure, but crew members have to remember a password. Admin resets via user management.

Pros

  • Stronger than PIN
  • Familiar pattern

Cons

  • Harder for field crew on mobile
  • Same build effort as A
C

Frontend-only role gates — no backend auth

Role and identity stored in localStorage. Backend routes stay open — no token validation. Fast to build, but any role can call any API endpoint if someone knows the URL. Fine for a fully trusted internal team on a closed network.

Pros

  • Fastest to build
  • No backend changes to routes

Cons

  • No real access control
  • Audit trail is unverified
