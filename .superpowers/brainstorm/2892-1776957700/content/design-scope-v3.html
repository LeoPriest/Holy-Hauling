In scope for this sub-project

✅ User model + DB migration — users table, bcrypt credential_hash, 4 roles, default admin seeded
✅ Auth endpoints — POST /auth/login (returns JWT), GET /auth/me
✅ JWT middleware — all existing routes require a valid token; current_user injected into every request
✅ Role enforcement
  • Admin only: /admin/users (user management), PATCH /settings (escalation thresholds, quiet hours, contact info)
  • Admin + facilitator: all lead operations, intake, AI review, notes
  • Supervisor only: PATCH /jobs/{id}/status (en route / started / complete)
  • All roles: GET /settings (read-only), GET /jobs, GET /auth/me
✅ Admin user management screen — /admin/users: list, add, edit role, reset PIN, deactivate
✅ Settings screen gating — facilitator sees settings read-only (no Save button); admin sees full editable form
✅ Login screen — username + PIN keypad, JWT stored in localStorage, auto-redirect on load
✅ Role-based routing — admin/facilitator → lead queue; crew/supervisor → /jobs
✅ GET /jobs + /jobs screen — booked leads filtered by role (crew sees limited fields; supervisor sees all)
✅ Light / dark mode toggle — preference stored in localStorage, applied app-wide, toggle in app header
✅ actor field populated from JWT — all existing audit trail events now carry the real username automatically
✅ Push notifications (Web Push) — browser-based, no native app required
  • Backend generates a VAPID key pair (stored in .env); exposes POST /push/subscribe to save a device's push subscription per user
  • Triggers: new job assigned to crew/supervisor → notify them; job escalated → notify supervisor
  • Frontend registers a service worker on login, requests notification permission, sends subscription to backend
  • Uses pywebpush on the backend to deliver Web Push payloads
  • Permission request shown once after login — skipped gracefully if denied

Testing plan

Backend (pytest)
  • Login correct PIN → 200 + JWT
  • Login wrong PIN → 401
  • Login deactivated user → 401
  • Request without token → 401
  • Facilitator hits /admin/users → 403
  • Facilitator hits PATCH /settings → 403
  • Admin hits PATCH /settings → 200
  • Crew hits PATCH /jobs/{id}/status → 403
  • GET /jobs returns only booked leads
  • Crew response omits phone + quote
  • Supervisor response includes phone + quote
  • Default admin seeded on empty DB
  • POST /push/subscribe saves subscription
  • Push fires when lead marked booked
Frontend (manual — dev server)
  • No token → login screen shown
  • Bad PIN → error message shown
  • Admin/facilitator login → lead queue
  • Crew/supervisor login → /jobs
  • Facilitator sees settings read-only
  • Admin sees full editable settings form
  • Crew job detail hides phone + quote
  • Supervisor sees status action buttons
  • Admin sees /admin/users nav item
  • Dark mode toggle persists on reload
  • Expired token → redirect to login
  • Push notification received when job assigned
